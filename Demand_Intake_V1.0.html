User Story — Demand Intake Agent

AI Studio Delivery Command Center — MVP Requirement Detail

User Story ID: DM-001 Version: 1.0 Date: July 2026 Status: Draft for Product Review
Scope of this document:
  • Defines the MVP user story for a Foundry-hosted Demand Intake Agent.
  • Covers two customer-facing channels: Microsoft Teams and embedded WebChat in the WebApp.
  • Focuses on direct customer demand submission, adaptive guided intake, validation, duplicate detection, customer confirmation, record creation, status visibility, and audit logging.
  • Excludes downstream scoring, governance triage, CPQ creation, routing, and delivery allocation except where required as integration handoff points.

2. Business Context

The Demand Intake Agent is the customer-facing front door for the AI Studio Delivery Command Center. The MVP requires customers to raise demand directly through the platform rather than relying on account teams to relay requests. The agent must guide the customer through intake, adapt the question flow to the demand type, validate completeness, perform semantic duplicate detection, and create a governed demand item that becomes visible in the customer's tracker with status Submitted.

This story supports the wider Demand Management module by creating a clean, structured demand item that can then be scored, prioritised, triaged, and routed by NTT DATA users. The customer must only see safe customer-facing fields and must never see internal scoring, commercial detail, capacity data, or governance rationale.

6. Main User Journey

The intake journey must work consistently across Teams and WebChat. Channel-specific user experience may differ, but the business outcome, captured data, validation rules, duplicate logic, and created demand record must be the same.

User opens agent→ Agent verifies context→ User describes request→ Agent classifies demand type→ Guided questions→ Completeness check→ Duplicate check→ Customer confirms→ Demand record created→ Status: Submitted

7. Detailed Functional Requirements

FR-001 — Agent availability across two channelsThe same Demand Intake Agent capability must be available in Microsoft Teams and embedded WebChat.
Channel access Customer-facing
The customer user must be able to start a new intake conversation from either Microsoft Teams or the AI Studio Delivery Command Center WebApp. The WebChat should be embedded within the customer portal and should recognise the logged-in user's organisation context. The Teams channel should support customer users who have been provisioned to access the agent through approved tenant or external access setup.
Expected behaviour: A user who starts in Teams and a user who starts in WebChat are guided through equivalent intake logic. The created demand record must have the same data structure regardless of channel.
FR-002 — User authentication and organisation contextThe agent must only allow authorised customer users to submit demand for their own organisation.
Identity and access control Security
Before a demand record is created, the platform must resolve the user's identity, organisation, permitted business units, and customer role. If the user's identity or organisation cannot be resolved, the agent must not create a demand item. It should provide a controlled message instructing the user to contact their platform administrator or NTT DATA support contact.
Security behaviour: The agent must not ask the user to manually type their organisation as the sole source of truth if authenticated organisation context is available. User-provided organisation text can be captured for clarification, but record ownership must come from the platform identity context.
FR-003 — Natural language demand captureThe user can begin by describing their need in plain language.
Initial request capture Agent-led
The agent must accept a free-text description such as "We want an AI tool to review supplier contracts" or "We need additional data engineering capacity for our chatbot build". The agent should extract an initial summary, potential demand type, business area, likely outcomes, and missing information.
In practice: The user writes a rough request. The agent responds with a short acknowledgement, proposes a demand type, and asks the next best question rather than forcing the user into a long static form.
FR-004 — Demand type classificationThe agent adapts the question flow based on the type of demand.
Demand typeExamplesAdditional question focus
AI use case / new solutionNew AI assistant, document intelligence tool, predictive model, workflow automation idea.Business problem, target users, expected value, process impacted, data availability, timeline, sponsor.
Enhancement to existing solutionImprove existing chatbot, add features to current analytics product, expand model coverage.Existing solution name, current limitation, requested change, affected users, urgency, production impact.
Capacity requestNeed additional developers, architects, data engineers, testers, or delivery support.Required roles, skill areas, start date, duration, expected commitment, location or shore preference.
Exploratory / unclear requestCustomer has a broad idea but does not know if it is a use case, enhancement, or capacity ask.Clarification questions; agent may keep classification as Unknown until enough information is captured.

FR-007 — Completeness validation and draft handlingIncomplete requests must not become unactionable records.
Draft and validation handling Manual + Auto
The agent must validate whether mandatory fields are present before creating a demand item. If required information is missing, the agent should continue asking focused questions. If the user cannot complete the submission, the agent should save the conversation as a draft demand intake where supported, mark the draft as incomplete, and allow the user to resume later.
Expected behaviour: The platform must not create a submitted demand item with no business area, no description, or no timeline. Incomplete items remain draft and are visible only as draft/intake-in-progress where customer visibility rules allow.
FR-008 — Semantic duplicate detectionThe agent checks for similar existing demand items before final submission.
Duplicate detection Agent-led
Before creating a new demand item, the agent must search existing demand records using semantic similarity, not keyword matching alone. The search must include demand items from the same customer organisation and, where access rules permit, other business units within that organisation. The agent must identify exact duplicates, likely duplicates, and related-but-distinct demand items.
Agent behaviour: The agent must never auto-merge demand records. It should show a customer-safe summary of potential matches, ask whether the new request is the same or distinct, and record the user's choice. Similar but distinct items should be cross-referenced on creation.
Duplicate resultCustomer-facing behaviourRecord behaviour
High-confidence duplicateAgent shows customer-safe summary of existing item and asks whether the user wants to follow existing item or continue as distinct.If user confirms same initiative, no new demand item is created unless business rules require a linked submission record.
Possible duplicateAgent explains that a similar request exists and asks the user to confirm same vs distinct.If distinct, new demand item is created with semantic duplicate reference and flagged for governance review.
Related demandAgent shows that related activity exists but does not block submission.New demand item is created with a related-demand cross-reference.
No matchAgent proceeds to final summary and confirmation.New demand item is created normally.
FR-009 — Customer review and confirmationThe user must confirm the structured demand summary before submission.
Submission confirmation Customer-facing
Before creating the demand item, the agent must present a concise summary containing title, demand type, business area, objective, expected value, proposed timeline, known risks or constraints, and duplicate/related-demand outcome. The user must be able to edit incorrect fields or confirm submission.
Confirmation options: Confirm and submit; edit a field; add more detail; cancel; save as draft.
FR-010 — Demand record creationA confirmed submission creates a governed Demand Item record.
Record creation Automatic
Once the customer confirms the summary, the platform must create a Demand Item record in the data layer. The record must include the captured business fields, customer organisation, submitter ID, channel of submission, duplicate references, generated summary, raw conversation reference, status, and audit metadata.
Initial status: The internal status should be created as Submitted or equivalent configured backend status. The customer-facing status label must display as Submitted.
FR-011 — Customer acknowledgement and tracker visibilityThe customer immediately sees the submitted item.
Acknowledgement Customer-facing
After record creation, the agent must confirm successful submission and provide the customer with the demand reference ID, title, submitted date/time, current customer-facing status, and where to track it. In WebChat, a link to the demand tracker should be shown. In Teams, a card or message should provide the same safe summary and link if enabled.
Customer sees: Reference ID, title, status Submitted, high-level summary, and next-step expectation such as "NTT DATA will review this request." Customer must not see score, scoring breakdown, governance notes, CPQ fields, rates, margin, or capacity data.
FR-012 — Agent action loggingEvery material agent action must be auditable.
Audit and traceability Governance
The agent must write entries to the Agent Action Log for conversation start, identity resolution, demand type classification, questions asked, answers received, completeness checks, duplicate search execution, duplicate results shown, customer confirmation, record creation, and any error or fallback event.
Audit behaviour: Logs must support traceability without exposing unnecessary customer-sensitive data to users who are not authorised to view it.

11. Non-functional Requirements

CategoryRequirement
SecurityAll access must respect customer and NTT DATA role-based access controls. Customer users must only create and view demand within their authorised organisation scope.
PrivacyConversation transcripts, attachments, and extracted summaries must be handled according to approved data retention and privacy policies.
ReliabilityThe agent must not create duplicate records due to retry, refresh, or repeated confirmation events. Record creation must be idempotent where technically feasible.
AvailabilityThe intake capability should remain available in at least one channel if the other channel is temporarily unavailable, subject to platform architecture.
PerformanceCommon agent responses should feel interactive. Duplicate search and record creation should return clear progress messages if they take longer than expected.
ExplainabilityThe agent should explain why it is asking for key fields in plain language, especially value, timeline, and business area.
AuditabilityAll material actions must be traceable through Agent Action Log and linked to the demand item or draft.
AccessibilityWebChat must support keyboard navigation, readable contrast, and screen-reader-friendly labels where applicable.
BrandingThe WebChat and confirmation components should follow the AI Studio / NTT DATA Command Center visual language used by the platform.

12. Error and Edge Case Handling

CaseExpected handling
User abandons conversationSave as draft if enough context exists and draft capability is enabled; otherwise log abandoned session without creating a demand item.
User provides vague requestAgent asks clarifying questions and may classify as exploratory or unknown until enough detail is gathered.
User asks for unsupported actionAgent explains that it can help submit demand and redirects to supported intake path.
User tries to view internal scoring or commercial detailAgent refuses to expose internal information and provides customer-safe status information only.
Duplicate search unavailableAgent should not silently skip duplicate detection unless product rules permit. It should either retry, save draft, or create with a flag that duplicate check failed and needs governance review.
Record creation failsAgent must not claim the item was submitted. It should show retry/support guidance and log the failure.
User changes channel mid-processIf conversation state is linked to the user and draft, the user should be able to resume. If not supported, agent should start a new intake and avoid exposing previous channel data unsafely.
Multiple users submit similar demandSemantic duplicate logic should surface possible matches and capture whether the request is same, related, or distinct.
User enters sensitive or regulated dataAgent should capture a sensitivity flag and avoid unnecessary repetition of sensitive content in summaries.

14. Sample Conversation

Example — New AI use case
Customer: We want to build an AI tool for contract review.
Agent: I can help capture that as a new AI Studio demand item. Is this a new AI use case, an enhancement to an existing solution, or a request for additional delivery capacity?
Customer: New AI use case.
Agent: Great. Which business area owns this request?
Customer: Legal Operations.
Agent: What problem are you trying to solve?
Customer: Contract review takes too long and we want to identify risky clauses faster.
Agent: What benefit are you expecting?
Customer: Faster review times and fewer missed compliance issues.
Agent: Is there a target timeline or business deadline?
Customer: We would like a discovery to start in September.
Agent: I found a similar request from another business unit in your organisation: "Automated legal document analysis" submitted three weeks ago. Is your request the same initiative or a separate but related request?
Customer: Separate, but related.
Agent: Thanks. Here is the demand summary I will submit...
Customer: Confirm and submit.
Agent: Submitted. Your demand reference is DEM-000123. Current status: Submitted. You can track it in your demand tracker.
